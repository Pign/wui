# CLI Reference

The `wui` CLI tool scaffolds, builds, runs, and cleans wui projects. Install it via haxelib:

```bash
haxelib install wui
```

Then invoke as:

```bash
haxelib run wui <command> [options]
```

Or, if you add haxelib's bin to your PATH:

```bash
wui <command> [options]
```

---

## Commands

### init

```bash
wui init [name]
```

Scaffold a new wui project. Defaults to `MyApp` if no name is given.

Creates:

```
<name>/
  src/
    <name>.hx        # main App class
  build.hxml          # Haxe build configuration
  wui.json            # project metadata
```

The generated `build.hxml` includes `-lib wui` and registers the `WinUIGenerator` macro. The generated source extends `wui.App` with a minimal `body()`.

**Example:**

```bash
wui init TodoApp
cd TodoApp
wui run
```

---

### build

```bash
wui build [options]
```

Compile the project through the full pipeline:

| Step | What happens |
|------|--------------|
| **1/4** Haxe compilation | Runs `haxe build.hxml`. The WinUIGenerator macro emits C++/WinRT source into `build/winui/`. |
| **2/4** hxcpp build | Compiles the hxcpp output to a static library in `build/cpp/`. |
| **3/4** NuGet restore | Restores Windows App SDK, CppWinRT, and SDK Build Tools packages into `build/packages/`. |
| **4/4** MSBuild | Builds the `.vcxproj` to produce a native `.exe`. |

Output: `build/winui/<Config>/<AppName>.exe`

Requires `wui.json` and `build.hxml` in the current directory.

---

### run

```bash
wui run [options]
```

Build (if needed) then launch the `.exe`. Accepts the same options as `build`.

---

### clean

```bash
wui clean
```

Delete the entire `build/` directory (hxcpp output, generated C++/WinRT, NuGet packages, and compiled binaries).

---

### help

```bash
wui help
```

Print usage information and list all commands.

---

### version

```bash
wui version
```

Print the wui version (currently `0.1.0`).

---

## Options

Options can be passed to `build` and `run`:

| Option | Description | Default |
|--------|-------------|---------|
| `--release` | Build with the Release configuration (optimized, no debug symbols). | Debug |
| `--arch=ARCH` | Target architecture. Values: `x64`, `arm64`. | `x64` |
| `--verbose`, `-v` | Show full MSBuild output instead of minimal. | off |

**Examples:**

```bash
wui build --release
wui build --arch=arm64
wui run --release --verbose
wui build --release --arch=arm64 -v
```

## wui.json

Every wui project has a `wui.json` in its root. This file configures the build.

```json
{
    "appName": "MyApp",
    "packageName": "com.haxe.myapp",
    "displayName": "My App",
    "windowsAppSdkVersion": "1.5.*",
    "targetFramework": "net8.0-windows10.0.22621.0",
    "architecture": "x64"
}
```

### Fields

| Field | Type | Description |
|-------|------|-------------|
| `appName` | string | Internal name. Used for the `.vcxproj`, `.exe`, and C++ namespaces. |
| `packageName` | string | Reverse-domain package identifier. |
| `displayName` | string | Human-readable name shown in the title bar (if not overridden by `appName()`). |
| `windowsAppSdkVersion` | string | Windows App SDK NuGet version constraint. |
| `targetFramework` | string | Target framework moniker. |
| `architecture` | string | Default build architecture (`x64` or `arm64`). Overridden by `--arch`. |

## build.hxml

The Haxe build file. Generated by `wui init`, but you can edit it freely.

```hxml
-cp src
-lib wui
--macro wui.macros.WinUIGenerator.register()
-main MyApp
-cpp build/cpp
```

Key entries:

- `-lib wui` -- pulls in the wui library from haxelib.
- `--macro wui.macros.WinUIGenerator.register()` -- registers the code generation macro that runs after Haxe compilation.
- `-cpp build/cpp` -- hxcpp output directory.

You can add additional `-cp`, `-lib`, or `-D` flags as needed.
